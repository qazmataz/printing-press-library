## Non-negotiable conventions (apply to EVERY command you implement)

1. RunE shape (verify-friendly):
   ```go
   RunE: func(cmd *cobra.Command, args []string) error {
       if len(args) == 0 && cmd.Flags().NFlag() == 0 { return cmd.Help() }
       if dryRunOK(flags) { fmt.Fprintln(cmd.OutOrStdout(), "would <do X>"); return nil }
       // required-input checks -> `_ = cmd.Usage(); return usageErr(fmt.Errorf(...))`
       // ... real work ...
   }
   ```
   Commands with NO required inputs: omit the usageErr branch. Commands whose only inputs are flags with defaults: the help-only branch still applies.
2. The generated stubs in internal/cli/*.go already declare the constructor names (newNovelXxxCmd) and are wired into root.go / parent commands. KEEP the constructor names. Replace the body. Change the second header line from "Generated by CLI Printing Press ... DO NOT EDIT." to "Hand-authored novel feature for appmagic-pp-cli." (keep the copyright line).
3. Output: build a typed view struct and finish with `printJSONFiltered(cmd.OutOrStdout(), view, flags)` so --json/--select/--compact/--csv/--quiet work. Initialize slices with `make([]T, 0)` so empty renders as [] not null.
4. Annotations: stubs already carry `"mcp:read-only": "true"` — keep. Add a `// pp:data-source <auto|local|live>` comment line directly above the `cmd := &cobra.Command{` literal (exact values listed per command below). Commands tagged `local` must reject `--data-source live` with a clear "no live equivalent" error; `live` must reject `--data-source local` ("no local data source"); read `flags.dataSource`.
5. Official-API calls: `c, err := flags.newClient()` then `c.Get(ctx, path, params)` / `c.Post(ctx, path, body)` with `ctx := cmd.Context()` (client handles timeout). `c.Post` returns `(json.RawMessage, int, error)`. Wrap errors with `classifyAPIError(err, flags)`.
6. Web-surface calls (web * commands only): `wc, err := webapi.New(flags.timeout)` (import "appmagic-pp-cli/internal/source/webapi"), wrap with `boundCtx(cmd.Context(), flags)` + `defer cancel()`. BEFORE any network: `if cliutil.IsVerifyEnv() { fmt.Fprintln(cmd.OutOrStdout(), "would query the appmagic.rocks web surface"); return nil }`.
7. Long-running fan-outs: `if cliutil.IsDogfoodEnv() { /* curtail to <=3 items / 1 page */ }`.
8. SQLite access: `--db` StringVar flag, default "" -> `defaultDBPath("appmagic-pp-cli")`. Missing-mirror guard for `local` data-source commands (per template: stat dbPath; if missing, stderr hint + `[]` for --json/--agent + return nil). Open with `store.OpenWithContext(ctx, dbPath)`; call `db.EnsureAppmagicTables(ctx)` right after open (it lazy-creates chart_snapshots, soft_launch_sightings, watchlist, entitlement_probes — see internal/store/appmagic_migrations.go for columns). NULL-safe scans: sql.NullString/NullInt64 for any nullable column.
9. Helpers that EXIST (do not reinvent): dryRunOK, boundCtx, usageErr/notFoundErr/authErr/apiErr/rateLimitErr, classifyAPIError, printJSONFiltered, filterFields, truncate, hintIfUnsynced/hintIfStale (sync_hint.go), defaultDBPath, cliutil.ParseDurationLoose (accepts 7d/1w), cliutil.IsVerifyEnv, cliutil.IsDogfoodEnv, resolveUnitedApp / unitedAppsByIDs / parseUnitedApps / stripStorePrefix / resolveTagID (appmagic_resolve.go), webapi.New/Get/Post (internal/source/webapi).
10. Duration flags: StringVar + cliutil.ParseDurationLoose. Never time.Duration flags.
11. Examples: `Example: strings.Trim(\`\n  appmagic-pp-cli <cmd> ...\n  appmagic-pp-cli <cmd> ... --json\n\`, "\n")` with 2-space indent, realistic args (Royal Match, com.dreamgames.royalmatch, TikTok, US, match-3).
12. Enrich every flag description (>=5 words, domain-specific). Stub flags say "TODO: describe" — replace all. Add any missing flags the design requires; kebab-case names.
13. Long descriptions: where the design gives a "Long:" text, set cmd.Long to it verbatim.
14. Tests: replace the t.Skip stub in the matching _test.go with REAL table-driven tests of the pure logic (parsing, diffing, median, delta computation) using fixture JSON. No network in tests. Minimum 2 real test funcs per owned test file. `go test ./internal/cli/ -run <YourTests>` must pass.
15. Parallel fan-outs that aggregate MUST keep per-item errors (fetchResult{idx,id,entry,err}), exclude failures from denominators, stderr-warn "N of M fetches failed; computed over K items", and include `fetch_failures` in the JSON envelope (omitempty).
16. Scan caps: any command scanning lists adds `--max-scan-pages` (or per-design cap flag), `scanned_<unit>` in envelope, and a `note` on zero-match-with-cap-hit naming the widening flag.

## Endpoint facts (verified against the live spec; param names are the generated CLI flag spellings where given)

- GET /united-applications?search=&limit= -> bare array of united apps: {id:int, name?, applications:[{name, publisher_name, ...}], store_ids?...}. Use resolveUnitedApp.
- POST /united-applications/search-by-ids with bare array body [ids...] -> same shape. Use unitedAppsByIDs.
- GET /tops/united-applications?sort=top_free|top_grossing|top_featuring&store=1..5&country=US&date=YYYY-MM-DD&aggregation=daily&limit=&tag_id= -> bare array [{place:int, united_application_id:int, value:int}].
- GET /tops/soft-launches?include_countries=&date_from=&date_to=&sort=current_app_downloads&store=&limit= -> bare array [{store_application_id, store:int, release_date, current_app_downloads, current_app_revenue, publisher_apps_count, publisher_countries_list[]}]. NOTE: no united id, no app name in rows.
- POST /applications/search-by-ids with bare array of store-app-id strings -> app metadata (name, publisher_name) for enrichment.
- GET /history/united-applications?... -> bare array [{united_application_id:int, date, country, store:int, downloads:int, revenue:int, featuring, top_free, top_grossing, created_at}]. CHECK exact query param names in the spec file (research dir below) or via the generated `history get-united-applications --help`.
- POST /retention-v2 body {store_application_id:string, store:int, country:string, date_start, date_end} -> {retention_1:[[ts,val]...]? shape is arrays of [number,number] pairs; treat each retention_N as series of [epoch_or_index, value] pairs; compute the latest/mean non-null value defensively}.
- GET /aso/by_app is POST /aso/by_app (CHECK method + body via spec/`aso get-by-app --help`): keyword positions per app per date. GET /aso/series?app_id=&countries=&terms=&aggregation=day -> {data:[{key:{appId,country,store,term}, points:[...]}]}.
- GET /live-ops/live-ops -> bare array [{game_name, description, duration:[...], live_ops_id, group_id, ...}]. CHECK its query params via `live-ops get --help` (likely store_application_id / united id / dates).
- Official spec JSON: /Users/<operator>/printing-press/.runstate/gd-skills-ff965ca1/runs/20260610-094753-319aefaa/research/appmagic-openapi3.json (grep it for exact param names instead of guessing).
- Web surface (webapi client, UNOFFICIAL, Bearer APPMAGIC_WEB_TOKEN):
  - GET /top/hourly-apps?topDepth=&store=&country=&date=YYYY-MM-DD (store 1/2/3 only, NOT 5) -> {data:[{top_free:{application,diff,rank...}, top_grossing:..., top_paid:...}], date}
  - POST /monetization-intelligence/offers {"store":1,"country":"US","limit":20,"dateFrom":"YYYY-MM-DD","dateTo":"YYYY-MM-DD"} -> {priceRange, totalCount, data:[{game_name, category, type, structure, duration, offers[], offers_calendar[]}]}
  - POST /feature-intelligence/live-ops/count-by-tags {"store":1,"country":"US","dateFrom":"...","dateTo":"..."} -> flat {tag_name: count} dict
  - GET /tags/apps-count?store=&country=&date= -> {data:[{tags:[tag_id], count:int}]}
  - Web param casing is exactly as shown (camelCase dateFrom/dateTo, topDepth).

## Per-command specs

### A1: chart-diff (internal/cli/chart_diff.go + chart_diff_test.go) and soft-launch-radar (internal/cli/soft_launch_radar.go + soft_launch_radar_test.go)

chart-diff — pp:data-source auto.
Flags: --sort (free|grossing|featuring; map to top_free/top_grossing/top_featuring; required), --store int (default 5... NOTE official tops accepts 1,2,3,4,5 per spec enum), --country (default "WW"), --date (optional capture date YYYY-MM-DD; default = latest official data date), --from/--to (explicit snapshot dates to diff), --top int (default 100, max 1000 -> limit param), --db, --no-capture bool.
Behavior: unless --no-capture or --data-source local: fetch GET /tops/united-applications for (sort, store, country, date=--date or today-2d), store rows into chart_snapshots (snapshot_date = the date param used), then enrich names for NEW/DROPPED apps only via unitedAppsByIDs (cap 50 ids). Diff: pick two snapshot dates (latest two distinct snapshot_date for the filter, or --from/--to); compute entered (in B not A), dropped (in A not B), moved (rank delta != 0, report top movers by |delta|). View: {sort, store, country, from, to, entered:[{rank, united_application_id, name}], dropped:[...], moved:[{...,from_rank,to_rank,delta}], note}. <2 snapshots: note explains and names the command to run again tomorrow; empty arrays; exit 0.
Long: "Use this command to see rank movement, new entrants, and dropouts between two synced top-chart snapshots for any chart sort. Do NOT use it to track newly soft-launched titles across test markets; use 'soft-launch-radar' instead. Do NOT use it for market-size aggregates by tag; use 'tag-rollup' instead."
Tests: pure diff function over fixture snapshot rows (entered/dropped/moved correctness, zero-diff case, single-snapshot note case).

soft-launch-radar — pp:data-source auto.
Flags: --countries CSV (default "PH,CA,AU"), --since string (default "30d", ParseDurationLoose), --store int (default 5), --publisher (substring filter), --tag (reserved: filter via tag_ids param if present), --limit int (default 50), --db, --no-capture.
Behavior: unless --no-capture/local: for each country (cap 5; dogfood cap 1): GET /tops/soft-launches {include_countries: country, date_from: today-90d, date_to: today, sort: current_app_downloads, store, limit}; upsert soft_launch_sightings keyed (country, store, store_application_id as united_application_id column — store the store_application_id string; first_seen=existing or today, last_seen=today). Enrich names for NEW sightings via POST /applications/search-by-ids (cap 50). Report: sightings WHERE first_seen >= now - since (+publisher filter), ordered first_seen desc, limit. View rows: {app: name, store_application_id, store, country, publisher_name, release_date, first_seen, downloads}. Include fetch_failures per country (rule 15) and scanned_countries.
Long: "Use this command to find newly detected soft-launch titles with first-seen dates per test market, optionally filtered by publisher or tag. Do NOT use it for general top-chart rank movement; use 'chart-diff' instead."
Tests: first-seen upsert logic + since-window filter over fixtures.

### A2: entitlements (internal/cli/entitlements.go + entitlements_test.go)

pp:data-source auto. Flags: --refresh bool, --db, --groups CSV (default all).
Probe map (group -> cheap GET): applications:/applications?search=tiktok&limit=1; united applications:/united-applications?search=tiktok&limit=1; publishers:/publishers?search=king&limit=1; united publishers:/united-publishers?search=king&limit=1; tops:/tops/united-applications?sort=top_free&store=2&country=US&date=<recent>&limit=1; charts: POST /dau minimal body (SKIP probe—POST with fabricated body is noisy; instead probe history:/history/united-application?... CHECK params, or mark charts group verdict "unknown(post-only)"); history:/history/united-applications (minimal params per spec); steam:/steam/last-date; period comparison:/period-comparison/last-date; categories:/categories; tags:/tags; aso:/aso/dates (CHECK params; if requires store/country use store=1&country=US); asa:/asa/dates; live-ops:/live-ops/games?limit=1 (CHECK); sdkint:/sdkint/sdks?limit=1 (CHECK GET vs POST; sdks/all also exists); Ad Intelligence:/adint/filter_values; contacts:/contacts/companies/1 (acceptable probe: GET by id; 404 also proves group reachable -> verdict included); featuring:/featuring?<params per spec, minimal>; keywords:/keywords/all; last date:/last-date.
For POST-only groups (charts), verdict "post-only" with detail "group has no cheap GET; probed indirectly via history". Verdicts: 200/422/400->included (validation/data reached = entitled); 403->not-included; 401->auth-required (if ALL groups 401: error message "credentials missing or invalid" but still write cache + exit 0 with verdict map); 404->included (routed past auth); 429->rate-limited; network->error. Persist to entitlement_probes (probe_method/path/http_status/verdict/checked_at). Without --refresh and cache fresh (<7d): read from cache, no network. View: {checked_at, groups:[{group, verdict, http_status, detail}], summary:{included, not_included, unknown}}.
Probes run sequentially with 100ms gaps (be polite); dogfood: probe only categories,tags,last date.
Tests: verdict classification function (status->verdict table) + cache-freshness logic.

### A4: tag-rollup (tag_rollup.go + test), aso-movers (aso_movers.go + test), liveops-overlay (liveops_overlay.go + test)

tag-rollup — pp:data-source auto. Flags: --tags CSV required, --country (default WW), --store (default 5... CHECK tops enum allows 5), --metric (downloads|revenue, default revenue), --period (default "30d"), --top int (default 20 apps per tag; dogfood 3), --db.
Behavior: resolve each tag via resolveTagID (needs synced taxonomy; hintIfUnsynced on "tags"); per tag: GET /tops/united-applications {sort: top_grossing (revenue) | top_free (downloads), tag_id, store, country, date:recent, limit:top} -> ids; then GET /history/united-applications for those ids over period (CHECK param names; batch if endpoint accepts multiple ids, else per-app fan-out with rule 15) -> sum metric per tag. View: {period, country, metric, tags:[{tag, tag_id, apps_counted, total}], fetch_failures, note}. Scan cap: --top IS the cap; document in help. Sum is an estimate-of-top-N: name it "top_n_total" + note text saying it sums the top N apps, not the whole market.
Long: "Use this command for aggregate market sizing (summed downloads or revenue) across one or more tags. Do NOT use it to list which apps moved within a chart; use 'chart-diff' instead."
Tests: tag aggregation over fixture history rows; note/cap behavior.

aso-movers — pp:data-source live. Args: app store-id or name positional required. Flags: --country (default US), --store (default 1), --since (default "7d"), --dataset (aso|asa, default aso), --limit int (default 25), --db (only for resolve fallbacks; optional).
Behavior: determine store_application_id: if arg looks like store id (contains '.' or all digits) use as-is; else resolveUnitedApp -> pick store id matching --store kind. Pull keyword positions at two dates: d2=latest available (today-2d), d1=d2-since. Endpoint: POST /aso/by_app or /asa/by_app (CHECK body via spec: grep '"/aso/by_app"' -A30 in the openapi3 json; likely {app_id/appIds, country/countries, date, store, limit}). Compute movers: gained (in d2 not d1), lost (in d1 not d2), moved (position delta), sorted by |delta| desc, cap --limit. View: {app, dataset, country, from_date, to_date, gained:[{term, position, score?}], lost:[...], moved:[{term, from_position, to_position, delta}]}.
Tests: mover-delta computation over fixture term lists (gained/lost/moved/no-change).

liveops-overlay — pp:data-source live. Args: app name/id positional required. Flags: --country (default "WW"), --since (default "90d"), --window int days (default 7, the before/after delta window), --db (optional).
Behavior: resolve app; fetch live-ops events: GET /live-ops/live-ops (CHECK params via spec — likely store_application_id or game filter + dates; live-ops covers a curated game list: if app not covered, return note "app not in the live-ops coverage list" + suggest 'live-ops get-games'); fetch GET /history/united-applications for the app over since+window; for each event with a start date inside window: avg daily downloads/revenue in [start-window, start) vs [start, start+window) -> delta pct.
